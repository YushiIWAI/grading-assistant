## 今回の変更（36回目）

### OCR精度改善: 高解像度画像抽出 + 領域クロップ

#### 1. PDF埋込画像の直接抽出 (`pdf_processor.py`)
- `_try_extract_page_image()`: スキャンPDFの埋込JPEG/PNGを直接抽出する新関数
- `pdf_to_images(prefer_embedded=True)`: 埋込画像が高解像度ならラスタライズせず使う
- 効果: 2377x1700px → 2429x3395px（約2倍の情報量）
- ページ回転の自動補正あり、小さい画像（ロゴ等）のフォールバックあり

#### 2. 領域クロップユーティリティ (`pdf_processor.py`)
- `crop_regions_from_image()`: 正規化bbox座標(0-1)でページ画像から解答欄を切り出す
- パディング付き、座標範囲外の自動補正

#### 3. レイアウト分析に bbox 座標を追加 (`scoring_engine.py`)
- `LAYOUT_REGION_SCHEMA`: `bbox` フィールド追加（optional）
- `build_layout_analysis_prompt()`: bbox座標（正規化矩形）を要求するプロンプトに変更
- Gemini は安定して bbox を返す（数回の実験で確認済み）

#### 4. OCRでクロップ画像送信 (`scoring_engine.py`)
- `GeminiProvider.ocr_student()`: マスク済み画像からbbox領域をクロップし、全体画像と一緒に送信
- `AnthropicProvider.ocr_student()`: 同様の実装
- `build_ocr_prompt_with_layout(has_region_crops=True)`: 拡大画像優先読み取りの指示を追加
- クロップが小さすぎる場合（ページ面積の5%未満）は自動スキップ
- クロップ付きの場合は `max_output_tokens=16384`, `thinking_budget=4096` に引き上げ

### バグ修正・堅牢性向上

#### 5. `_extract_json()` の修復ロジック強化
- `dict` / `list` が直接渡された場合のハンドリング追加
- 閉じ ` ``` ` がない場合（出力切れ）でもパースを試みる
- 未閉じの文字列・配列・オブジェクトを自動補完して再パース
- `_api_call_with_retry()`: 失敗時に生レスポンスの先頭1000文字をログ出力

#### 6. Gemini タイムアウト延長
- `GeminiProvider.TIMEOUT`: 120秒 → 180秒（高解像度+クロップ画像で処理が重くなるため）

#### 7. Streamlit 起動時の軽微なバグ修正 (`app.py`)
- 関数外の `return` を `st.stop()` に置換（Python構文エラー）
- `st.slider` の `min_value == max_value` エラー回避（学生1名の場合）
- `list_sessions()` の認証なし呼び出しを `try/except` で保護

## OCR実験結果（スキャンテスト.pdf）

### 実験条件
- PDF: A4横、手書き答案（魯迅「故郷」）、1ページ
- プロバイダ: Gemini 3.1 Pro Preview
- ルーブリック: 簡易（問1: 記述、問2: 記述）

### 精度比較
| 項目 | 改善前（200dpi ラスタライズ） | 改善後（埋込画像+クロップ） |
|------|---------------------------|--------------------------|
| 画像解像度 | 2377x1700px | 2429x3395px |
| 「瓜畑」 | 瓜畑 | **色瓜畑**（より正確） |
| 「閏土」 | 関谷（誤り） | **閏土**（正しい） |
| 「ヤンおばさん」 | ばさん（欠落） | **やンおばさん**（ほぼ正確） |

### 判明した課題

#### パフォーマンス問題（最重要）
- レイアウト分析: ~30-60秒/回（初回のみ、キャッシュ可能）
- OCR: ~60-120秒/回/学生（thinking model のため）
- **学年200名の場合: 200×2分 = 約7時間** ← 現実的ではない
- 対策案:
  - `gemini-2.5-flash` での速度テスト（精度とのトレードオフ）
  - レイアウト分析は初回のみ（既に実装済み）
  - バッチ並列化（現在は逐次処理）
  - thinking_budget の削減（OCRに深い推論は不要？）
  - クロップ画像のみ送信し全体画像を省略（API負荷軽減）

#### レイアウト分析の不安定さ
- 同じ画像でもbbox精度が毎回変動する
- 簡易ルーブリックの設問記述が曖昧だと、実際の答案構造との対応付けに失敗する
- 正しいルーブリック（麻布「故郷」など）を使えば安定するかは未検証

#### JSON出力切れ
- thinking model が出力トークンの大部分を思考に使い、本文JSONが途中で切れる
- `max_output_tokens` を16384に引き上げたが、長文記述式では依然リスクあり
- JSON自動補完ロジックで緩和（未閉じ文字列/括弧を補完）
