# Grading Assistant 商用化・実用プロダクト化ロードマップ

## この文書の目的

この文書は、grading-assistant を「個人利用の便利ツール」から、
学校現場で継続利用される実用プロダクトへ育てるための判断材料をまとめたもの。
今後のエージェントAIは、機能追加や設計変更を提案する際に、
この文書をプロダクト方針の基準として参照すること。

作成日: 2026-03-12
改訂日: 2026-03-13（5チームレビュー統合）

## 現状の総評

現状の grading-assistant は、採点精度を改善し続けるための土台が既にある
「強いプロトタイプ」である。

特に強い点:

- Streamlit UI が教員の実際の採点フローにかなり寄っている
- OCR、横断採点、検証パス、needs_review などの実務に効く仕組みがある
- ゴールドデータ評価とユニットテストがあり、改善ループを回せる

一方で、広く導入される有料プロダクトになるための最大の課題は、
採点ロジックの細部よりも、信頼できる運用基盤を整えることにある。

### 現場教員からの評価（レビューより）

- 横断採点と needs_review は教員が最も助かる機能。「怪しいところだけ見る」ワークフローへの期待が高い
- review_reason（なぜ確認が必要か）の表示は、ピンポイントで判断できるため好評
- 「AIが仮採点、最終判断は教員」は教員として唯一受け入れられるアプローチ。保護者への説明責任を果たすために必須
- 採点基準レビューフェーズ（AIが質問を生成）は「事前に答えるより、採点しながら基準を固める」のが実態のため、現在の「事前質疑応答」形式は優先度が低い。**ただし、AIに実際の答案をざっと読み込ませ、ありそうなモデルケース（「こういう言い回しの場合はどうする？」）を抽出してもらい、それに答えていく形式であれば実用的**（開発者フィードバック）
- ルーブリックのYAML入力は現実的でない。自然言語で書いた基準をAIが構造化すべき

## 最終的に目指すプロダクト像

目標は「AIが自動で採点するサービス」ではない。

目指すべき価値は次の通り:

- 教員が、根拠付きで、短時間に、安全に採点を確定できる
- AIは仮採点と要確認抽出を担い、最終判断は教員が持つ
- 学校が安心して導入できるレベルのデータ保護・監査性・説明可能性を備える

つまり、このプロジェクトの主役は AI ではなく、
「教員の採点判断を支えるレビュー基盤」である。

### 差別化の核（レビューより）

- 「記述採点の精度」と「教員レビューフロー」の組み合わせ。大手プラットフォームが後追いしにくい領域
- needs_review、横断採点、教員修正履歴は現場密着型の機能。早期にユーザーフィードバックで磨くべき
- 差別化の本質は「採点ロジック」ではなく「教員のワークフローへの深い統合」にある

## 非目標

当面は次を目標にしない:

- 全教科・全校種への一気の拡大
- 完全自動採点による教員確認の省略
- 料金設計より先に営業・拡販を進めること
- 入試採点への早期参入（社会的・制度的抵抗が極めて強い。定期テスト・実力テストで実績を作ってから）

まずは「高校の定期テスト・国語・記述問題」で強く勝ち、
その後に対象範囲を広げる方が成功確率が高い。

### ターゲット戦略（レビューより）

- 「国語記述」の範囲が広すぎる。高校入試と中間テストと小論文では採点の性質がまったく異なる
- 最初のターゲット: **私立中高一貫校の国語科教員**。予算の自由度が高く、意思決定が速い
- 横展開先: 社会科（地歴公民）の記述問題。国語と同様の構造で、比較的容易に対応可能
- 入試は中長期目標。まず定期テスト・実力テストで実績を作る

## 早期に検証すべき仮説（レビューより追加）

商用化の前提となる仮説を、フェーズ0で検証すること:

1. **需要仮説**: 教員は記述採点の支援に月額いくらまで払うか？
2. **精度仮説**: 現状の採点精度で教員は「使える」と判断するか？ 特に実際の高校生の手書き答案（崩し字、薄い鉛筆、消しゴム跡）での精度
3. **コスト仮説**: 1答案あたりのAPI原価はいくらか？ 40人クラス×5問の記述採点で何円かかるか？ 採算ラインはどこか？
4. **導入仮説**: 学校（特に管理職・教育委員会）はAIへの答案画像送信を許容するか？
5. **継続仮説**: 1回使った教員は次の試験でも使うか？
6. **確認負荷仮説**: needs_reviewで絞り込んだ結果、確認作業は「ゼロから採点するより明らかに速い」か？ 600個中80個程度の確認で済むレベルか？

## 優先度の高い課題

### 0. 外部AI送信のデータ保護（最優先・即時対応）

**全レビューチームが最大のリスクとして指摘。フェーズ1以前に設計を確定すべき。**

答案画像には氏名・筆跡・解答内容が含まれ、個人情報保護法上の「個人情報」に該当する。
外部AIサービス（Gemini API）への送信は、以下の法的・倫理的リスクを伴う:

#### 法的リスク
- 2022年改正個人情報保護法:
  - **外国にある第三者への提供（第28条）**: Google（米国企業）のAPIに送信する場合、本人への情報提供義務が発生
  - **個人データの取扱いの委託（第25条）**: 委託先の監督義務。Gemini APIの利用規約がこの義務を果たせるか精査が必要
  - **利用目的の特定と通知（第17条・第21条）**: 「採点のためにAIに送信する」ことを保護者・生徒に事前に明示する必要がある
- 未成年の個人情報は特に慎重な扱いが求められる。保護者の同意取得（オプトイン）が必要

#### 即時対応事項
- **API送信前の氏名マスキング機能の設計・実装**: 現在のコードは答案画像を氏名付きのままGemini APIに送信している。これは即時対処すべき
- **Gemini APIのDPA（Data Processing Addendum）確認**: 送信データがモデル学習に使用されないことの契約上の担保（有料APIとFree tierで扱いが異なる場合がある）
- **データの所在地（リージョン）確認**: 日本国内処理が求められる場合がある

#### フェーズ1に含めるべき対応
- 外部AI送信のデータフロー図の作成
- API利用規約の法務レビュー
- 保護者同意取得フローの設計
- プライバシーポリシー・個人情報取扱規程のドラフト
- インシデント対応計画（漏洩発覚時の72時間以内速報義務への対応手順）

### 1. 認証・権限・学校単位のデータ分離

現状は単一アプリ・単一運用に近く、
広く導入するには次の概念が不足している:

- 学校 / 組織
- ユーザー
- 権限ロール（管理者、教員、閲覧のみ等）
- データ所有者
- 担当クラスや担当試験の境界

今後は、誰がどの答案と採点結果にアクセスできるかを
明示的に制御できる設計へ移行する必要がある。

現在の共通パスワード認証（`check_password()`）は商用に耐えない。
**文科省ガイドラインでは多要素認証（MFA）の導入が推奨**されている。

### 2. データ保護・保存期間・削除の設計

このプロダクトは答案画像、OCR結果、氏名、得点、教員メモを扱う。
商用化するなら、技術設計として最低限以下が必要:

- 保存データの暗号化（at rest）
- 通信経路暗号化（in transit）
- 学校単位のデータ分離（マルチテナント: RLSかスキーマ分離かの判断が必要）
- 保存期間と自動削除
- 手動削除
- エクスポート（学校解約時の全データエクスポート・完全削除手順を含む）
- アクセスログ（誰がいつどの答案を閲覧・修正したか。改ざん防止を含む）
- 監査ログ（ログ自体の保存期間と保護方針）
- バックアップと災害復旧（暗号化・保存場所・復旧テストの設計）
- データの所在地（国内保存の要否）

#### 画像ストレージの容量見積もり（レビューより追加）
- 答案画像は 1クラス40人 × 数ページで数百MB。学校単位で年間数GB
- S3/GCS等のオブジェクトストレージの選定とコスト見積もりがフェーズ1に必要

### 3. Streamlit 一体型からサービス分離へ

現状はプロトタイプとして適切だが、
有料プロダクトとしては UI、業務ロジック、保存、AI実行が密結合している。

目標構成の例:

- フロントエンド: 教員向け採点UI
- API / アプリケーション層: 試験、答案、採点run、レビュー操作の管理
- ワーカー: OCR、横断採点、検証パス
- DB: ユーザー、学校、試験、採点結果、監査イベント
- オブジェクト保存: PDF、画像、エクスポート成果物

この分離により、可用性、再実行性、監査性、複数ユーザー運用が成立する。

#### アーキテクチャ移行の方針（レビューより追加）
- **フェーズ1の冒頭でAPI層（FastAPI等）を先に切り出す**。Streamlitは当面フロントとして残すが、直接scoring_engineを呼ぶのではなくAPI経由にする
- Streamlitのセッションステートに認証情報を載せる設計はプロダクション品質にならない。API層に認証を実装すれば、フロント差し替え時の影響を最小化できる
- フェーズ1の最初のタスクとして**ADR（Architecture Decision Record）**を書く: DB選定（PostgreSQL推奨）、テナント分離方式、APIフレームワーク選定を文書化
- DBマイグレーション戦略（Alembic等）: 学校運用中にダウンタイムなしでスキーマ変更する手段を設計
- デプロイ先（クラウド / オンプレ）とデータ所在地要件を早期に決定

### 4. バックグラウンドジョブ化

OCR や採点は今後必ず長時間処理になる。
商用運用では、UI セッションに処理をぶら下げたままにしないこと。

必要な要素:

- ジョブキュー
- 非同期実行
- 途中失敗からの再開
- 部分成功の記録
- 進捗表示
- タイムアウトと再試行の管理

教員がブラウザを閉じても処理が継続し、
後から結果を確認できる構成が必要。

#### 補足（レビューより）
- 答案数が多い場合（40人×4ページ=160画像）、Streamlitのセッションタイムアウトに引っかかる可能性がある。フェーズ1後半でCelery/RQ等の導入を検討
- ジョブキュー（Celery + Redis / Cloud Tasks等）の選定はデプロイ先に依存する。インフラ選定を先に確定すべき

### 5. 採点runの再現性と監査証跡

学校導入では「なぜこの点なのか」を後で説明できる必要がある。
そのためには、採点結果だけでなく実行条件も保存する必要がある。

保存すべき情報:

- 使用モデル
- プロンプト版
- ルーブリック版
- 実行日時
- 実行者
- 教員による修正履歴
- AI初期値と最終確定値の差分

採点は「値」だけでなく「判断の履歴」を扱う設計へ移行すること。

### 6. 評価基盤のプロダクト化

evaluation/ の仕組みはこのプロジェクトの大きな強みであり、
今後も中核に据えるべきである。

今後の拡張方針:

- 教科・学年・問題形式ごとのゴールドデータ拡充
- OCR 精度の独立評価
- needs_review の Precision / Recall の継続監視
- モデル変更時の自動比較
- リリース前の回帰テスト自動化
- **「AIを使った方が採点が安定する」というエビデンスの取得**（レビューより: 採点者間ばらつき vs AI支援時のばらつきの比較。最強の営業材料になる）

新機能やモデル変更は、
必ず評価指標の改善または維持を確認してから導入すること。

### 7. 商用提供のためのコスト管理と課金基盤

現状は利用者が API キーを入力する前提だが、
有料プロダクト化する場合は、運営側が API コストを管理する構成が基本になる。

必要な論点:

- 1学生あたり / 1答案あたりの原価
- OCR と採点のコスト分解
- 学校単位の利用量可視化
- 上限設定
- プラン設計（**学校は年度単位の定額制で予算を取る。従量課金は予算申請と合わない**）
- 請求処理
- Gemini APIのレートリミット: 複数学校が同時に採点を走らせた場合の制御

BYO API key は将来的な例外オプションとしてはあり得るが、
標準形としては運営管理型の方が整合的。

**API原価の試算は即座に実施すべき。採算が合わない場合、ビジネスモデル自体が成立しない。**

### 8. 学校導入の意思決定プロセスへの対応（レビューより追加）

技術基盤だけでなく、学校の導入承認プロセスに対応する資料の準備が必要:

- **セキュリティチェックシート**: 学校・教育委員会向け。外部AI送信の安全性を説明
- **導入説明資料**: 管理職・教務主任向け。費用対効果、個人情報保護方針を含む
- **保護者向け説明文書**: AIによる採点支援の利用と、データの取り扱いについて
- **文科省「教育情報セキュリティポリシーに関するガイドライン」準拠チェックリスト**

公立校は教育委員会の一括導入決裁が必要で時間がかかる。
まずは私立学校の個別導入を狙うべき。

### 9. 現場ニーズの見落とし（レビューより追加）

レビューで指摘された、ロードマップに欠けていた現場ニーズ:

- **採点結果の集計・返却資料生成**: 頻出の誤答パターン、正答率等を集計し、模範解答と講評の下書きを生成。教員にとって非常に価値が高い
- **採点の「ゆれ」検出**: 同じような解答に異なる点数が付いていないかのチェック。横断採点の延長として実装すべき
- **答案のソート・グルーピング**: 同じような解答をまとめて表示し一括採点。横断採点の思想と合致
- **既存採点基準（Word等）の取り込み**: 毎回YAML入力は非現実的。自然言語をAIが構造化すべき
- **途中保存と再開**: 200枚を一度に終わらせることは稀。翌日続きから再開できることが必須
- **スキャン手順ガイド**: 学校の複合機での設定方法、画質、両面、ファイル名の指針。スマホ撮影（歪み・影・解像度）への対応も検討
- **印刷・紙出力**: 答案画像に点数を重畳して印刷する機能
- **年度切り替え**: 4月のクラス替え・担当替えに対応するデータ管理
- **校務支援システム連携**: CSVだけでなく、C4th/EDUCOM/スズキ校務等にインポート可能な形式
- **オフライン/ローカル動作の選択肢**: クラウド送信を禁止している学校向け。少なくとも検討項目に入れるべき
- **模範解答・解答例の蓄積**: 「こういう解答は何点」の具体例を蓄積・共有。採点基準すり合わせ時間を短縮
- **赤ペン添削（コメント返却）との連動**: 記述採点では点数だけでなく「ここがもう少し書けていれば加点」等のフィードバックを答案に書く学校が多い
- **採点基準すり合わせワークフロー**: 複数教員で模範答案を見ながら基準を固める工程。「試し採点→基準調整→本採点」のワークフロー
- **採点基準レビューの改良**: 現在の「事前質疑応答」形式ではなく、**AIに実際の答案をざっと読み込ませ、判断が分かれそうなモデルケース（「この言い回しは基準Bに該当するか？」）を実例ベースで抽出し、教員がそれに答えていく形式**にすると実用的（開発者フィードバック）

## 法令・ガイドライン対応（レビューより追加）

### 個人情報保護法

- **利用目的の公表・通知（第17条・第21条）**: プライバシーポリシーの策定。答案データの取得・利用・AI送信の目的を明示
- **安全管理措置（第23条）**: 組織的・人的・物理的・技術的安全管理措置の具体設計
- **第三者提供の制限（第27条）**: Gemini APIへの送信が第三者提供にあたるかの整理
- **委託（第25条）**: Googleとの間で委託契約が必要な可能性
- **外国にある第三者への提供（第28条）**: 本人への情報提供義務（移転先の個人情報保護制度等の説明）
- **開示・訂正・利用停止請求（第33-35条）**: 保護者・生徒からの開示請求に応じる仕組み
- **漏洩等報告（第26条）**: 個人情報保護委員会への報告義務、本人通知義務への対応手順
- **個人情報保護影響評価（PIA）**: 法的義務ではないが、教育機関向けでは実施を強く推奨
- **個人情報取扱事業者としての体制整備**: 個人情報保護管理者の選任、従業者教育等

### 文科省ガイドライン

文部科学省「教育情報セキュリティポリシーに関するガイドライン」（令和6年改訂版）:

- **クラウドサービス利用時の要件**: データの国内保存、アクセス制御、暗号化、監査証跡
- **校務系と学習系の分離**: 答案・成績は「校務系」に分類。より高いセキュリティレベルが要求される
- **外部サービス連携時のリスク評価**: 外部AIサービスへのデータ送信について、学校のセキュリティポリシーに基づく評価が必要
- **認証要件**: 多要素認証（MFA）の導入が推奨
- **アクセス制御**: 職務に応じた最小権限の原則
- **GIGAスクール構想との整合**: 学校のネットワーク環境（フィルタリング等）で動作することの確認

## 実装・設計の優先順位

### フェーズ0: Discovery（レビューより追加）

**フェーズ1の前に実施。1〜2ヶ月。**

目的: 商用化の前提となる仮説を検証し、フェーズ1以降の設計判断に必要な情報を揃える。

取り組むこと:

- **教員ヒアリング・プロトタイプ試用**: 3〜5人の現役教員（理想は私立高校の国語科主任）に現行プロトタイプで実際の定期テストを採点してもらう。「160枚中、修正が必要だったのはN枚」のような数字を取る
- **API原価の試算**: 40人クラスの記述採点1回あたりの原価を実データで算出
- **OCR精度の実地検証**: テストデータではなく、実際の高校生の手書き答案で精度を確認
- **Gemini APIのDPA確認**: データ処理規約の取得、学習利用の有無、データ保存期間・処理場所の把握
- **パイロット校2〜3校との関係構築**: ICTに積極的な私立高校が理想
- **WTP（支払い意思）ヒアリング**: 「月額いくらなら払うか」を教員・管理職に確認
- **文科省ガイドライン準拠チェックリストの作成**
- **API送信前の氏名マスキング機能の設計**: 技術的な実装方針を確定

### フェーズ1: 信頼基盤の整備

優先度: 最優先

#### フェーズ1a: API層切り出し + DB導入（レビューより分割）

取り組むこと:

- ADR（Architecture Decision Record）の作成: DB選定、テナント分離方式、APIフレームワーク選定
- デプロイ先（クラウド / オンプレ）とデータ所在地要件の決定
- ターゲットインフラ構成の1枚絵
- API層（FastAPI等）の切り出し: scoring_engine/storageをAPI背後に配置
- JSON保存からDB（PostgreSQL推奨）+ オブジェクト保存への移行
- DBマイグレーション戦略（Alembic等）
- API送信前の氏名マスキング機能の実装

#### フェーズ1b: 認証・マルチテナント・監査ログ

取り組むこと:

- ユーザー個別認証 + MFA
- ユーザー/学校/権限モデル
- マルチテナントデータ分離
- 監査ログ（改ざん防止を含む）
- 保存期間と削除設計（学校解約時の完全削除手順を含む）
- 保存時暗号化（at rest）+ 通信経路暗号化（in transit）
- プライバシーポリシー・利用規約のドラフト
- セキュリティチェックシート（学校・教育委員会向け）のドラフト
- インシデント対応計画の策定
- APIキーの管理をKMS等に移行

## 今後の AI への指示方針

今後のエージェントAIは、提案や実装の際に以下を優先すること:

1. 採点精度の改善より先に、学校導入に耐える信頼基盤を壊さないこと
2. 新機能は、評価可能性・監査可能性・データ保護に悪影響がない形で設計すること
3. 「AIの自動化」より「教員が安心して確定できる導線」を優先すること
4. 対象を広げる提案より、国語記述採点での勝ち筋を強くする提案を優先すること
5. **外部AI送信時のデータ保護（氏名マスキング等）を常に意識すること**
6. **「確認」の負荷が高すぎないか常に検証すること。120人分を全確認するなら自分で採点するのと変わらない**
