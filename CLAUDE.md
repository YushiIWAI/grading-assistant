## 再発防止ルール（過去に複数回再発したバグ）

### タブリセット禁止パターン（Streamlit）
- `st.tabs()` の **前** に条件付き表示（`if ... : st.markdown()` 等）を置いてはならない
- 状態変化で表示/非表示が切り替わるとウィジェットツリーがずれ、タブ選択がリセットされる
- 条件付きコンテンツはタブの **中** に配置すること
- `@st.fragment` 内で `st.rerun(scope="app")` を使わないこと（フラグメントスコープのリランで十分）

### Gemini thinking_budget=0 禁止
- thinking-only モデル（Gemini 3.1 Pro Preview 等）では `thinking_budget=0` は API エラーになる
- typed モードの高速化目的でも最低 `1024` を設定すること

## セッション管理
- セッション開始時: `.handoff.md` があれば読み込んで前回の続きから再開。なければ `doc-map.md` → `.claude/skills/project-context/skill.md` の順に読む
- セッション中: コンテキストが逼迫してきたら、ユーザーの指示を待たずに `doc-map.md` を参照して資料更新 + `.handoff.md` 作成 + ユーザーへ通知
- セッション終了時: `doc-map.md` に従い関連資料を更新し、`.handoff.md` に作業状態を記録する

## Gemini API呼び出し時のプロンプト指針（実装予定）

### 空間プロンプト（Spatial Prompting）
Geminiはテキストと画像を一つのネットワークで同時処理しており、文字の内容だけでなく画像内の空間的位置も高解像度で理解している。この特性を活かし、答案・文書の画像解析では**2段階の思考プロセス**を明示する:
1. **構造認識フェーズ**: まず画像全体のレイアウトを把握させ、解答欄・セクション等の領域を特定・報告させる
2. **局所OCRフェーズ**: 特定した各領域の内部のみに焦点を当てて読み取らせる。領域外のノイズ（汚れ・メモ等）は無視を指示

### Google Workspace CLI 連携（将来）
- Classroom APIを職場アカウントで再認証し、成績データ連携に活用予定

### Gemini thinking mode 制約（必須）
- Gemini 3.1 Pro Preview など thinking-only モデルでは `thinking_budget=0` は **絶対に使えない**（APIが400エラーを返す）
- typed モード等で高速化したい場合でも最低 `thinking_budget=1024` を設定すること
- 新しいモデルを追加する際は、そのモデルが thinking 必須かどうかを確認すること

### 2段構えシステム（設計方針）
答案用紙のレイアウトは学校・教科・教員によって異なるため、固定テンプレートでは対応できない。
- **1回目の呼び出し**: 答案画像の構造自体を認識させる（「このレイアウトを分析し、解答欄の位置と構成を報告して」）
- **2回目の呼び出し**: 1回目の結果を踏まえて空間プロンプトを動的に組み立て、各解答欄の内容を読み取る
- 同一形式の答案を繰り返し処理する場合は、初回の構造認識結果をテンプレとしてキャッシュし、2回目以降は局所OCRのみ実行する最適化も検討
