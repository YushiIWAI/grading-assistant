# 合成テストデータ仕様設計書

> 作成日: 2026-03-07
> 目的: 採点エンジンのチューニング・回帰テスト用ゴールドデータの設計

---

## 1. 全体構成

### 1.1 データ形式

テストデータはPDF生成とは分離し、**JSON形式のゴールドデータ**として管理する。
これにより、採点エンジンのテキスト入力→採点出力を自動評価できる。

```
evaluation/
  test-data-spec.md       ← 本ファイル（仕様）
  exam-research.md        ← 調査レポート（既存）
  gold/                   ← ゴールドデータ
    todai_gendaibun.json  ← 東大型現代文（既存テストの構造化版）
    azabu_chugaku.json    ← 麻布中型読解
    kokugo_kiso.json      ← 国語基礎（漢字・文法・短答）
  runner.py               ← 自動評価スクリプト
```

### 1.2 ゴールドデータのスキーマ

```json
{
  "exam_id": "todai_gendaibun",
  "exam_title": "現代文（東大型模擬試験）",
  "source_rubric": "rubrics/todai_rubric.yaml",
  "questions": [
    {
      "question_id": "2",
      "question_type": "descriptive",
      "max_points": 14,
      "description": "傍線部(ア)「聴くということは…」とはどういうことか、説明せよ。",
      "model_answer": "聴くとは…（模範解答テキスト）",
      "scoring_elements": [
        {"id": "A", "description": "「了解の枠内に回収する」の否定の言い換え", "points": 4},
        {"id": "B", "description": "「枠組みそのものが揺さぶられる」の説明", "points": 5},
        {"id": "C", "description": "「予期を超えた意味」への言及", "points": 3},
        {"id": "expr", "description": "文末・論理構成", "points": 2}
      ]
    }
  ],
  "test_cases": [
    {
      "case_id": "q2_full",
      "question_id": "2",
      "student_answer": "聴くとは相手の発言を…（解答テキスト）",
      "expected_score": 14,
      "score_tolerance": 1,
      "score_level": "full",
      "expected_elements": ["A", "B", "C", "expr"],
      "tags": ["boundary:none"],
      "rationale": "全要素を正確に含む模範的な解答"
    }
  ]
}
```

### 1.3 フィールド定義

| フィールド | 型 | 説明 |
|---|---|---|
| `case_id` | string | テストケースの一意ID |
| `question_id` | string | 対応する設問ID |
| `student_answer` | string | 生徒の解答テキスト（OCR後を想定） |
| `expected_score` | number | 期待する得点 |
| `score_tolerance` | number | 許容誤差（±N点以内なら合格） |
| `score_level` | enum | `full` / `high_partial` / `low_partial` / `zero` / `off_topic` |
| `expected_elements` | string[] | 含まれるべき採点要素のID |
| `tags` | string[] | 分類タグ（後述） |
| `rationale` | string | この得点になる理由（人間向けメモ） |

### 1.4 タグ体系

| タグカテゴリ | 値 | 説明 |
|---|---|---|
| `boundary` | `none` / `upper` / `lower` | ボーダーケースか |
| `error_type` | `surface_reword` / `missing_element` / `off_topic` / `factual_error` / `format_error` | 減点理由の分類 |
| `student_level` | `advanced` / `mid` / `struggling` / `minimal` | 想定する生徒のレベル |
| `azabu` | `true` | 麻布生の解答スタイルを反映 |

## 2. 問題タイプ別テストケース設計

### 2.1 短答問題（漢字・語句）

**カバーすべきケース:**

| score_level | 内容 | 期待得点 |
|---|---|---|
| `full` | 正確な漢字 | 満点 |
| `zero` | 似た漢字の混同（受容→需要） | 0点 |
| `zero` | 送り仮名の誤り | 0点 |
| `zero` | 空欄 | 0点 |

**注意点:**
- 短答は部分点なし。正誤の二択
- `score_tolerance: 0`（完全一致が必要）
- OCRエラーの検知能力もテスト対象（将来）

**テストケース数の目安:** 各問3〜4ケース（正答1 + 誤答パターン2〜3）

### 2.2 読解記述（短め: 60〜70字・2行論述）

**カバーすべきケース:**

| score_level | 内容 | 期待得点率 | tolerance |
|---|---|---|---|
| `full` | 全要素＋適切な文末 | 100% | ±1点 |
| `high_partial` | 核心要素あり、周辺1要素欠落 | 60-80% | ±2点 |
| `high_partial` | 全要素あるが表現が不正確 | 50-70% | ±2点 |
| `low_partial` | 方向性は合っているが日常語レベル | 20-35% | ±2点 |
| `low_partial` | 1要素のみ正確 | 25-35% | ±1点 |
| `zero` | 的外れ（設問の誤読） | 0-10% | ±1点 |
| `zero` | 白紙・極端に短い | 0% | 0 |

**「惜しい」解答の重点パターン（麻布生向け）:**
1. **概念は理解しているが本文のキーワードを使っていない** — 「自分の枠組みが揺さぶられる」を「考え方が変わる」と日常語化
2. **論理の方向性は合っているが因果の精度が甘い** — 原因と結果を混同、あるいは中間ステップを省略
3. **文末形式ミス** — 「〜ということ。」であるべきところを「〜と思う。」

**テストケース数の目安:** 各問6〜8ケース

### 2.3 読解記述（長め: 120字・4行論述）

**カバーすべきケース:**

| score_level | 内容 | 期待得点率 | tolerance |
|---|---|---|---|
| `full` | 全要素網羅、字数適切 | 100% | ±1点 |
| `high_partial` | 3/4要素、字数適切 | 70-80% | ±1点 |
| `high_partial` | 全要素あるが浅い | 50-60% | ±2点 |
| `low_partial` | 2/4要素 | 40-50% | ±2点 |
| `low_partial` | 表面的な要約のみ | 20-30% | ±2点 |
| `zero` | 自分の感想（本文の論を踏まえていない） | 0-10% | ±1点 |
| `zero` | 字数大幅超過（120字制限に200字以上） | 減点あり | ±1点 |
| `zero` | 白紙 | 0% | 0 |

**追加の検証ポイント:**
- 字数超過の減点処理が正しいか
- 「本文全体の趣旨を踏まえて」の条件を満たしているかの判定

**テストケース数の目安:** 各問7〜9ケース

### 2.4 心情説明問題

**東大型にはないが、麻布中の定期テストで頻出。追加シナリオとして設計する。**

| score_level | 内容 | 典型例 |
|---|---|---|
| `full` | 背景＋出来事＋心情を正確に記述 | 「秘密を知られ馬鹿にされたため、気が滅入っている」 |
| `high_partial` | 心情の方向性は合っているが表現がズレ | 「許せない気持ち」（正: うんざり） |
| `low_partial` | 出来事のみ、心情に言及なし | 「狸が笑ってきた」 |
| `zero` | 本文と異なる場面の内容 | 全くの的外れ |

### 2.5 理由説明問題（「なぜか」型）

| score_level | 内容 | チェックポイント |
|---|---|---|
| `full` | 因果関係を正確に、文末「〜から。」 | 全要素＋形式 |
| `high_partial` | 因果は正しいが1要素欠落 | 部分点の精度 |
| `low_partial` | 「深く考えているから」程度の抽象記述 | 抽象度ペナルティ |
| `zero` | 内容説明を書いている（設問タイプの誤読） | 設問タイプ判別 |

## 3. 得点レベルの定義

### 3.1 5段階レベル

| レベル | 英語ID | 得点率の目安 | 特徴 |
|---|---|---|---|
| 満点 | `full` | 90-100% | 模範解答に近い。全要素＋適切な表現 |
| 高部分点 | `high_partial` | 50-80% | 方向性は正しい。1-2要素の欠落or表現の不正確さ |
| 低部分点 | `low_partial` | 15-40% | 大筋は理解。日常語レベル、要素不足が顕著 |
| 0点 | `zero` | 0% | 白紙、字数大幅不足 |
| 的外れ | `off_topic` | 0% | 設問の誤読、本文と無関係、自分の感想のみ |

### 3.2 ボーダーケースの重点

採点エンジンのチューニングで最も重要なのは **`high_partial` と `low_partial` の境界**。

具体的に検証すべきパターン:
1. **キーワードはあるが論理が不正確** → high_partial の下限
2. **キーワードはないが論理の方向性は正しい** → low_partial の上限
3. **要素Aは完璧、要素Bは部分的** → 要素間の配点バランス

## 4. 麻布生の解答スタイル

### 4.1 特徴（exam-research.md + 実際の傾向）

- **語彙レベルが高い**: 中学生でも「自己認識」「多義性」等の抽象概念を使える生徒がいる
- **独自の読みを入れがち**: 模範解答にない角度からの解釈を書くことがある
- **記述量が多い**: 指定字数ギリギリまで書く傾向。白紙は少ない
- **「なんとなく正しい」記述**: 本文の雰囲気は掴んでいるが、核心概念を正確に踏まえていない

### 4.2 テストケースへの反映

- `advanced` レベルの生徒には本文のキーワードを正確に使った高度な解答を設定
- `mid` レベルでは「日常語への読み替え」パターンを重点的に
- `struggling` でも完全な白紙は少なく、「何か書いてある」が的外れ、というケースを設定
- 独自解釈の解答（模範解答と異なるが本文に根拠がある）を `high_partial` として含める

## 5. 試験シナリオ一覧

### Scenario 1: 東大型現代文（既存の拡張）

- ファイル: `gold/todai_gendaibun.json`

- 問題構成: 漢字3問 + 2行論述3問 + 120字論述1問

- 元データ: `generate_todai_test.py` の5名分を構造化 + 追加ケース

### Scenario 2: 麻布中型 読解テスト（新規）

- ファイル: `gold/azabu_chugaku.json`
- ルーブリック: `rubrics/azabu_chugaku.yaml`（新規作成）
- 問題構成: 書き抜き1問 + 心情説明1問 + 内容説明1問 + 理由説明1問 + 意見記述1問
- テストケース数: 約30件
- 題材: 架空の小説文（中学入試レベル）

### Scenario 3: 国語基礎（新規）

- ファイル: `gold/kokugo_kiso.json`
- ルーブリック: `rubrics/kokugo_kiso.yaml`（新規作成）

- テストケース数: 約20件
- 目的: 短答・選択問題の正誤判定精度の検証

## 6. 自動評価の基準

### 6.1 合格基準

| 指標 | 基準 |
|---|---|
| **スコア一致率** | 全ケースの80%以上が `score_tolerance` 内 |
| **短答正誤一致率** | 95%以上（漢字・選択の正誤判定） |
| **重大な誤採点なし** | `full` を `zero` に、または `zero` を `full` に採点するケースが0件 |
| **ボーダーケース** | `high_partial` / `low_partial` の50%以上が tolerance 内 |

### 6.2 評価レポートの出力

```
=== 採点エンジン評価レポート ===
試験: 東大型現代文
総テストケース: 35
合格: 29/35 (82.9%)
不合格: 6/35

--- 不合格ケース ---
q2_low_partial_2: 期待=3±2, 実際=7 (乖離: +4)
  解答: 「相手の話を聞くときに…」
  理由: 日常語レベルの記述に過大評価

--- 問題タイプ別 ---
short_answer: 12/12 (100%)
descriptive_2line: 14/18 (77.8%)  ← 要改善
descriptive_120: 5/7 (71.4%)     ← 要改善
```

## 7. 実装計画

### Phase 1: ゴールドデータ作成（本セッション）

1. `generate_todai_test.py` の5名分データを `gold/todai_gendaibun.json` に構造化
2. 各問にボーダーケースを追加（+15件程度）
3. 評価ランナー `runner.py` の基本実装

### Phase 2: シナリオ拡充（次回以降）

1. 麻布中型シナリオの作成
2. 国語基礎シナリオの作成
3. 実際の採点エンジンでの評価実行 + プロンプト調整

### Phase 3: 継続的チューニング

1. 実運用で見つかった誤採点ケースをゴールドデータに追加
2. 回帰テストとして定期実行
